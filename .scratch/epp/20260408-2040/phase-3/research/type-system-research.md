# Type System and Validation Architecture Research

## Problem Statement

Phase 1A is greenfield for domain types. The codebase currently has zero domain types -- only Supabase client setup, auth helpers, and env validation. Patterns established now propagate through 30+ tables and 50+ server actions. Wrong patterns create compounding type-safety debt. This research defines the exact discriminated union for `NodeWithDetails`, the `ActionResult<T>` pattern with error codes, the complete Zod schema inventory for Phase 1A, file organization, and tsconfig hardening.

## Recommended Solution

### 1. NodeWithDetails Discriminated Union

The estimate tree uses a hybrid base + detail table model (Section 3 of data architecture). The TypeScript type system must mirror this: a shared base type with a `nodeType` discriminant field that narrows to the correct detail shape.

```typescript
// ──────────────────────────────────────────────────────────────────
// File: src/lib/types/domain/nodes.ts
// ──────────────────────────────────────────────────────────────────

import type { Database } from "@/lib/types/supabase";

// ── Raw database row types (generated by supabase gen types) ─────
// These are the direct table row shapes. We re-export aliases for clarity.

type DbEstimateNode = Database["public"]["Tables"]["estimate_nodes"]["Row"];
type DbNodeItemDetails = Database["public"]["Tables"]["node_item_details"]["Row"];
type DbNodeAssemblyDetails = Database["public"]["Tables"]["node_assembly_details"]["Row"];

// ── Node type literal union ──────────────────────────────────────

export type NodeType = "group" | "assembly" | "item";

// ── Client visibility ────────────────────────────────────────────

export type ClientVisibility = "visible" | "hidden" | "summary_only";

// ── Base node fields (shared by all 3 types) ─────────────────────
// These come from the estimate_nodes base table.

export interface NodeBase {
  readonly id: string;
  readonly estimateId: string;
  readonly parentId: string | null;
  readonly path: string | null;
  readonly sortOrder: number;
  readonly name: string;
  readonly description: string | null;
  readonly phaseId: string | null;
  readonly costCodeId: string | null;
  readonly clientVisibility: ClientVisibility;
  readonly subtotal: number;
  readonly contingencyAmount: number;
  readonly overheadAmount: number;
  readonly totalPrice: number;
  readonly catalogSourceId: string | null;
  readonly catalogSourceType: "item" | "assembly" | null;
  readonly catalogVersion: number | null;
  readonly referenceName: string | null;
  readonly flagged: boolean;
  readonly createdAt: string;
  readonly updatedAt: string;
  readonly createdBy: string | null;
}

// ── Item detail fields ───────────────────────────────────────────

export interface ItemDetails {
  readonly qty: number;
  readonly rawQty: number;
  readonly qtyMode: "numeric" | "formula" | "ratio";
  readonly qtyFormula: string | null;
  readonly ratioNumerator: number | null;
  readonly ratioDenominator: number | null;
  readonly unitId: string | null;
  readonly unitCost: number;
  readonly costFormula: string | null;
  readonly costType: "material" | "labor" | "equipment" | "subcontractor" | "other" | null;
  readonly contingencyRate: number;
  readonly overheadRate: number;
  readonly unitPrice: number | null;
  readonly wasteFactor: number;
  readonly minOrderQty: number | null;
  readonly packageSize: number | null;
  readonly packageUnitId: string | null;
  readonly bidType: "bid" | "allowance" | "estimate";
  readonly allowanceBudget: number | null;
  readonly allowanceStatus: "pending_selection" | "selected" | "finalized" | null;
  readonly vendorId: string | null;
  readonly instructions: string | null;
  readonly specifications: string | null;
  readonly archivedAt: string | null;
}

// ── Assembly detail fields ───────────────────────────────────────

export interface AssemblyDetails {
  readonly assemblyUnitId: string | null;
  readonly assemblyQty: number;
  readonly derivedUnitCost: number | null;
  readonly qtyFormula: string | null;
}

// ── Discriminated union ──────────────────────────────────────────
// The `nodeType` field is the discriminant.

export interface GroupNode extends NodeBase {
  readonly nodeType: "group";
  readonly details: null;
}

export interface AssemblyNode extends NodeBase {
  readonly nodeType: "assembly";
  readonly details: AssemblyDetails;
}

export interface ItemNode extends NodeBase {
  readonly nodeType: "item";
  readonly details: ItemDetails;
}

export type NodeWithDetails = GroupNode | AssemblyNode | ItemNode;

// ── Type guards ──────────────────────────────────────────────────

export function isGroupNode(node: NodeWithDetails): node is GroupNode {
  return node.nodeType === "group";
}

export function isAssemblyNode(node: NodeWithDetails): node is AssemblyNode {
  return node.nodeType === "assembly";
}

export function isItemNode(node: NodeWithDetails): node is ItemNode {
  return node.nodeType === "item";
}

// ── Tree node (with children, for client-side tree building) ─────

export interface TreeNode extends NodeWithDetails {
  readonly children: TreeNode[];
}

// ── Convenience: client visibility check ─────────────────────────

export function isClientVisible(node: NodeWithDetails): boolean {
  return node.clientVisibility !== "hidden";
}
```

**Key design decisions:**
1. **`nodeType` as discriminant** (not `node_type`) -- camelCase follows the codebase convention for TypeScript; the mapping from snake_case DB columns happens in the data access layer.
2. **`details: null` for groups** -- Groups have no detail table. Using `null` rather than `undefined` makes the discriminant exhaustive -- you must check `nodeType` before accessing `details`.
3. **`readonly` on all fields** -- Nodes are fetched from the server. Mutations go through server actions, not direct property assignment. This prevents accidental local mutation.
4. **Separate `ItemDetails` / `AssemblyDetails` interfaces** -- These are independently useful for form components and validation schemas.

### 2. Snapshot Data Typing

**Recommendation: Branded `Readonly` type, NOT a separate `FrozenNode` type.**

Rationale: Snapshot data has the exact same shape as live data (it IS a serialized estimate tree). A separate `FrozenNode` type would require maintaining two parallel type hierarchies that must stay in sync. Instead, use a branded type that prevents passing snapshot data where mutable data is expected, while sharing the same structural definition.

```typescript

export type SnapshotNode = Readonly<NodeWithDetails> & {

  readonly id: string;

```

**Why branded `Readonly` over separate `FrozenNode`:**
- Same structural shape -- no parallel hierarchy to maintain
- Brand prevents accidental use in mutation paths
- `Readonly` at the type level prevents property assignment
- Snapshot sub-records use slightly wider string types (e.g., `costType: string` instead of the full union) for forward compatibility -- if a new cost type is added, old snapshots still parse

### 3. ActionResult<T> with Error Codes

```typescript
// ──────────────────────────────────────────────────────────────────
// File: src/lib/types/action-result.ts
// ──────────────────────────────────────────────────────────────────

// ── Error code enum ──────────────────────────────────────────────
// Organized by domain. Client code can switch on these for
// specific error handling (retry, redirect, field highlighting).

export const ERROR_CODE = {
  // Auth / authorization
  UNAUTHORIZED: "UNAUTHORIZED",
  FORBIDDEN: "FORBIDDEN",
  ROLE_INSUFFICIENT: "ROLE_INSUFFICIENT",

  // Validation
  VALIDATION_ERROR: "VALIDATION_ERROR",
  INVALID_INPUT: "INVALID_INPUT",
  MISSING_REQUIRED_FIELD: "MISSING_REQUIRED_FIELD",

  // Not found
  NOT_FOUND: "NOT_FOUND",
  ESTIMATE_NOT_FOUND: "ESTIMATE_NOT_FOUND",
  NODE_NOT_FOUND: "NODE_NOT_FOUND",
  PROJECT_NOT_FOUND: "PROJECT_NOT_FOUND",
  SNAPSHOT_NOT_FOUND: "SNAPSHOT_NOT_FOUND",

  // Conflict / constraint violation
  CONFLICT: "CONFLICT",
  OPTIMISTIC_LOCK_FAILED: "OPTIMISTIC_LOCK_FAILED",
  DUPLICATE_NAME: "DUPLICATE_NAME",
  CONSTRAINT_VIOLATION: "CONSTRAINT_VIOLATION",

  // Tree operations
  INVALID_PARENT: "INVALID_PARENT",
  CIRCULAR_REFERENCE: "CIRCULAR_REFERENCE",
  MAX_DEPTH_EXCEEDED: "MAX_DEPTH_EXCEEDED",
  ITEM_HAS_CHILDREN: "ITEM_HAS_CHILDREN",

  // Snapshot operations
  SNAPSHOT_IMMUTABLE: "SNAPSHOT_IMMUTABLE",
  SNAPSHOT_RESTORE_BLOCKED: "SNAPSHOT_RESTORE_BLOCKED",
  SCHEMA_VERSION_MISMATCH: "SCHEMA_VERSION_MISMATCH",

  // Status / lifecycle
  INVALID_STATUS_TRANSITION: "INVALID_STATUS_TRANSITION",
  ESTIMATE_LOCKED: "ESTIMATE_LOCKED",

  // Server / database
  DATABASE_ERROR: "DATABASE_ERROR",
  INTERNAL_ERROR: "INTERNAL_ERROR",
  RATE_LIMITED: "RATE_LIMITED",
} as const;

export type ErrorCode = (typeof ERROR_CODE)[keyof typeof ERROR_CODE];

// ── ActionResult<T> ──────────────────────────────────────────────
// Discriminated union on `success`. Server actions ALWAYS return this.
// NEVER throw from a server action -- return ActionResult instead.

export type ActionResult<T = void> = ActionSuccess<T> | ActionError;

export interface ActionSuccess<T = void> {
  readonly success: true;
  readonly data: T;
}

export interface ActionError {
  readonly success: false;
  readonly error: string;           // User-facing message (safe to display)
  readonly code: ErrorCode;         // Machine-readable code for programmatic handling
  readonly fieldErrors?: Readonly<Record<string, string[]>>; // Per-field validation errors
}

// ── Factory functions ────────────────────────────────────────────
// Enforce consistent creation. Never construct ActionResult manually.

export function ok(): ActionResult<void>;
export function ok<T>(data: T): ActionResult<T>;
export function ok<T>(data?: T): ActionResult<T> {
  return { success: true, data: data as T };
}

export function err(
  error: string,
  code: ErrorCode,
  fieldErrors?: Record<string, string[]>,
): ActionError {
  return { success: false, error, code, ...(fieldErrors ? { fieldErrors } : {}) };
}

// ── Convenience error constructors ───────────────────────────────

export function notFound(entity: string): ActionError {
  return err(`${entity} not found.`, ERROR_CODE.NOT_FOUND);
}

export function unauthorized(): ActionError {
  return err("You must be signed in to perform this action.", ERROR_CODE.UNAUTHORIZED);
}

export function forbidden(): ActionError {
  return err("You do not have permission to perform this action.", ERROR_CODE.FORBIDDEN);
}

export function validationError(
  message: string,
  fieldErrors?: Record<string, string[]>,
): ActionError {
  return err(message, ERROR_CODE.VALIDATION_ERROR, fieldErrors);
}

export function optimisticLockFailed(): ActionError {
  return err(
    "This record was modified by another user. Please refresh and try again.",
    ERROR_CODE.OPTIMISTIC_LOCK_FAILED,
  );
}
```

**Key design decisions:**
1. **`const` object + derived type** instead of `enum` -- TypeScript enums have known issues (nominal typing breaks tree-shaking, reverse mapping is confusing). `as const` objects are the modern standard.
2. **`fieldErrors` on `ActionError`** -- Server actions validate with Zod. When validation fails, the per-field errors are passed through so form components can display inline errors.
3. **Factory functions** -- `ok()` and `err()` enforce consistent construction. Code reads as `return ok(project)` or `return notFound("Estimate")`.
4. **`void` default** for `ActionResult<T>` -- Mutations that don't return data (e.g., `deleteNode`) use `ActionResult` with no generic parameter.

```typescript
import type { ZodError } from "zod";
import { validationError } from "@/lib/types/action-result";
import type { ActionError } from "@/lib/types/action-result";

/**
 * Convert a ZodError into an ActionError with per-field error messages.
 */
export function formatZodError(error: ZodError): ActionError {
  const fieldErrors: Record<string, string[]> = {};

  for (const issue of error.issues) {
    const path = issue.path.join(".");
    if (!fieldErrors[path]) {
      fieldErrors[path] = [];
    }
    fieldErrors[path].push(issue.message);
  }

  const firstMessage = error.issues[0]?.message ?? "Invalid input.";

  return validationError(firstMessage, fieldErrors);
}
```

### 6. tsconfig.json Additions

Current `tsconfig.json` has `"strict": true` which enables:
- `strictNullChecks`
- `strictFunctionTypes`
- `strictBindCallApply`
- `strictPropertyInitialization`
- `noImplicitAny`
- `noImplicitThis`
- `alwaysStrict`

**Add these flags for Phase 1A:**

```jsonc
{
  "compilerOptions": {
    // ... existing options ...

    // ── New strict flags for Phase 1A ────────────────────────────
    "noUncheckedIndexedAccess": true,   // obj[key] returns T | undefined, not T
    "noImplicitReturns": true,          // Every code path must return a value
    "noFallthroughCasesInSwitch": true, // Require break/return in switch cases
    "exactOptionalPropertyTypes": true, // Distinguish between undefined and missing
    "noPropertyAccessFromIndexSignature": true, // Force bracket notation for index signatures
    "forceConsistentCasingInFileNames": true    // Prevent case mismatch issues on case-insensitive OS
  }
}
```

**Flag-by-flag rationale:**

| Flag | Why | Risk |
|------|-----|------|
| `noUncheckedIndexedAccess` | Prevents silent `undefined` access on arrays and records. Critical for JSONB data and tree traversal where array indices may be out of bounds. | Low -- codebase is small. A few `!` assertions may be needed on guaranteed-present lookups. |
| `noImplicitReturns` | Catches missing returns in complex switch/if chains. Critical for server actions that must always return `ActionResult`. | None -- this is a pure improvement. |
| `noFallthroughCasesInSwitch` | Prevents accidental fallthrough in `switch(node.nodeType)` exhaustive matches. | None -- intentional fallthrough is rare and can use explicit `// falls through` comment. |
| `exactOptionalPropertyTypes` | Distinguishes `{ x?: string }` (missing) from `{ x: string \| undefined }` (present but undefined). Critical for update schemas where omitted fields mean "don't change" and `undefined` means something else. | Medium -- may require adjusting some optional property patterns. Worth the precision for 30+ Zod schemas. |
| `noPropertyAccessFromIndexSignature` | Forces `obj["key"]` instead of `obj.key` for index signatures. Makes it clear when you're accessing a known property vs an arbitrary key. | Low -- mainly affects JSONB/Record usage. |
| `forceConsistentCasingInFileNames` | Prevents `import from './Nodes'` on Windows resolving to `./nodes.ts`. Catches deployment bugs where Linux (case-sensitive) would fail. | None -- already best practice. |

**Exact diff for tsconfig.json:**

```diff
  "compilerOptions": {
    "target": "ES2017",
    "lib": ["dom", "dom.iterable", "esnext"],
    "allowJs": true,
    "skipLibCheck": true,
    "strict": true,
+   "noUncheckedIndexedAccess": true,
+   "noImplicitReturns": true,
+   "noFallthroughCasesInSwitch": true,
+   "exactOptionalPropertyTypes": true,
+   "noPropertyAccessFromIndexSignature": true,
+   "forceConsistentCasingInFileNames": true,

```

## Trade-offs Considered

| Decision | Alternative | Why Rejected |
|----------|-------------|--------------|
| `nodeType` discriminant on domain type (not DB column name) | Use `node_type` to match DB | TypeScript convention is camelCase. The mapping layer handles snake_case -> camelCase. Consistent with Next.js/React ecosystem. |
| Branded `Readonly` for snapshots | Separate `FrozenNode` type hierarchy | Parallel types that must stay in sync. Any column added to `NodeWithDetails` must also be added to `FrozenNode`. Branded approach shares the structural definition. |
| `as const` object for ErrorCode | TypeScript `enum ErrorCode` | Enums have tree-shaking issues, reverse mapping confusion, and nominal typing that complicates testing. `as const` is the modern standard. |
| Zod discriminatedUnion for createNode | Single schema with conditional refinements | Discriminated union is type-safe at compile time. Refinements only validate at runtime. The 3 node types have genuinely different required fields (item needs details, group does not). |
| `exactOptionalPropertyTypes` | Skip it (it's the most breaking flag) | Without it, `undefined` values in update payloads could accidentally clear fields. The precision is worth the migration cost, especially with a small existing codebase. |

## Test Cases

### Type System Tests (compile-time -- verified by `tsc --noEmit`)

1. **Discriminated union narrowing:** Verify that `switch(node.nodeType)` narrows `details` correctly in all 3 branches.
2. **Snapshot brand prevents mutation path usage:** Verify that `SnapshotNode` cannot be passed to a function expecting `NodeWithDetails` without explicit cast.
3. **ActionResult exhaustive handling:** Verify that `if (!result.success)` narrows to `ActionError` with `code` and `fieldErrors`.
4. **Error code completeness:** Verify all error codes are valid `ErrorCode` values via satisfies.

### Validation Schema Tests (runtime -- vitest)

5. **createNodeSchema rejects item without details:** Pass `{ nodeType: "item", ... }` without `details` field -- expect validation failure.
6. **createNodeSchema accepts group without details:** Pass `{ nodeType: "group", ... }` without `details` -- expect success.
7. **createNodeSchema rejects item as root:** Pass `{ nodeType: "item", parentId: null }` -- expect failure (items must have a parent).
8. **Ratio denominator cannot be zero:** Pass `{ ratioDenominator: 0 }` in item details -- expect validation failure.
9. **UUID validation rejects non-UUID strings:** Pass `"not-a-uuid"` for any ID field -- expect failure.
10. **Estimate update requires version:** Pass update without `version` field -- expect failure (optimistic locking).
11. **PIN must be exactly 6 digits:** Pass `"12345"` (5 digits), `"1234567"` (7 digits), `"abcdef"` (letters) -- all expect failure.
12. **formatZodError produces correct field error shape:** Pass a ZodError with multiple issues on different paths -- verify output shape.
13. **ok() factory returns correct discriminant:** Verify `ok(data).success === true`.
14. **err() factory returns correct discriminant:** Verify `err("msg", "NOT_FOUND").success === false`.
15. **Default values applied correctly:** Verify `createProjectSchema.parse({ name: "Test" })` has `status: "lead"`.
