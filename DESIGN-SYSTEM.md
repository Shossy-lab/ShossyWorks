# ShossyWorks Design System

## Foundational Rule (NON-NEGOTIABLE)

**Zero hardcoded styles.** Every color, spacing value, font size, border radius, shadow, and visual property MUST come from a design token or CSS variable. No exceptions. No "just this once." No "it's temporary."

If a token doesn't exist for what you need, **add it to the system first**, then use it.

This rule applies to:
- All components (including "placeholder" and "temporary" UI)
- All pages and layouts
- All interactive states (hover, focus, active, disabled)
- All responsive breakpoints

---

## Design Philosophy

**Clean. Modern. Minimalistic. Intentional.**

Every visual choice must be deliberate. Default AI output ("generic AI slop") is explicitly rejected. The UI should feel like it was designed by a human with opinions, not generated by a model with defaults.

### Core Principles

1. **Shapes are honest.** A rectangle is a rectangle. A square is a square. Do not round their corners. If something is meant to be round, make it a pill (fully rounded ends) or a circle (perfect circle). There is no in-between.

2. **Less is more.** Remove before adding. If a visual element doesn't serve a clear purpose, it doesn't belong.

3. **Data first, UI second.** Architecture and data correctness take priority. But when UI is needed, do it right from the start — don't ship placeholder styles that will need rework.

4. **Consistency through tokens.** Every visual decision is encoded in the token system. Deviating from tokens is a bug.

---

## Shape Language

| Shape | Border Radius | Use For |
|-------|--------------|---------|
| Rectangle / Square | `--radius-none` (0) | Cards, panels, containers, inputs, text areas, modals, dropdowns |
| Pill | `--radius-full` (9999px) | Primary action buttons, tags, badges, status indicators |
| Circle | `--radius-full` (9999px) + equal width/height | Avatars, icon buttons, status dots |

**Forbidden:** `rounded-sm`, `rounded-md`, `rounded-lg`, `rounded-xl` on containers. These produce the generic look we're avoiding.

## CSS Custom Properties (Design Tokens)

All tokens are defined in `src/app/globals.css` and consumed via Tailwind utility classes or direct `var()` references.

### Colors

```css
:root {
  /* Background */
  --color-bg-primary: #ffffff;
  --color-bg-secondary: #f5f5f5;
  --color-bg-tertiary: #e5e5e5;
  --color-bg-inverse: #0a0a0a;

  /* Surface (cards, panels) */
  --color-surface: #ffffff;
  --color-surface-hover: #fafafa;
  --color-surface-active: #f5f5f5;

  /* Border */
  --color-border: #e5e5e5;
  --color-border-strong: #d4d4d4;
  --color-border-focus: #0a0a0a;

  /* Text */
  --color-text-primary: #0a0a0a;
  --color-text-secondary: #525252;
  --color-text-tertiary: #a3a3a3;
  --color-text-inverse: #fafafa;
  --color-text-link: #0a0a0a;

  /* Interactive */
  --color-interactive: #0a0a0a;
  --color-interactive-hover: #262626;
  --color-interactive-text: #fafafa;

  /* Semantic */
  --color-success: #16a34a;
  --color-warning: #ca8a04;
  --color-error: #dc2626;
  --color-info: #2563eb;

  /* Semantic backgrounds (subtle) */
  --color-success-bg: #f0fdf4;
  --color-warning-bg: #fefce8;
  --color-error-bg: #fef2f2;
  --color-info-bg: #eff6ff;
}
```

## Tailwind Integration

Tokens are extended into Tailwind via `@theme` in `globals.css` so they're usable as utility classes:

```css
@theme {
  --color-bg-primary: var(--color-bg-primary);
  --color-surface: var(--color-surface);
  /* ... mapped to Tailwind utilities */
}
```

### Semantic Class Patterns

| Instead of | Use |
|-----------|-----|
| `bg-white` | `bg-[var(--color-bg-primary)]` or semantic class |
| `bg-gray-50` | `bg-[var(--color-bg-secondary)]` |
| `text-gray-900` | `text-[var(--color-text-primary)]` |
| `text-gray-600` | `text-[var(--color-text-secondary)]` |
| `border-gray-300` | `border-[var(--color-border)]` |
| `rounded-lg` | remove (use `rounded-none` or `rounded-full`) |
| `bg-blue-600` | `bg-[var(--color-interactive)]` |
| `text-blue-600` | `text-[var(--color-text-link)]` |
| `bg-red-50` | `bg-[var(--color-error-bg)]` |
| `text-red-700` | `text-[var(--color-error)]` |

## Component Patterns

### Buttons

```
Primary:   pill shape, solid fill, inverse text
Secondary: pill shape, border only, primary text
Ghost:     no border, subtle hover background
Icon:      circle, transparent background, hover fill
```

### Inputs

```
Sharp corners (radius-none), 1px border, clear focus ring
Focus: border transitions to --color-border-focus
```

### Cards / Panels

```
Sharp corners, 1px border, no shadow by default
Optional: subtle shadow on hover for interactive cards
```

### Sidebar Navigation

```
Collapsible (expanded ↔ icon-only)
Active item: left accent border, subtle background fill
Hover: subtle background fill
No rounded corners on nav items
```

---

## Forbidden Patterns

These are explicitly banned. If you see them in a component, they are bugs:

1. `rounded-sm`, `rounded-md`, `rounded-lg`, `rounded-xl` on containers/cards/inputs
2. Direct Tailwind color classes (`bg-blue-600`, `text-gray-900`) — use tokens
3. Hardcoded hex colors in component files
4. Hardcoded `px` values for spacing — use spacing tokens
5. Inline `style=` attributes for visual properties
6. Magic numbers for width/height — use layout tokens or responsive utilities
7. `shadow-md`, `shadow-lg` — use token-based shadow values

---

## Adding New Tokens

When a new visual property is needed:

1. Add the CSS custom property to `globals.css` under the appropriate section
2. Document it in this file
3. Use it in the component via `var(--token-name)` or Tailwind arbitrary value
4. Never skip this process — "I'll add the token later" is not acceptable
